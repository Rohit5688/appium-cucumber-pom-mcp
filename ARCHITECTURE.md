# Appium Cucumber POM MCP Server: Architecture & Internals

This document details the architectural decisions, intelligent systems, internal workflows, and integration points for the **Appium Cucumber POM MCP Server**.

---

## 🏗️ System Architecture

This MCP Server is built on a **Modular Service-Oriented Chassis**, strongly typed using TypeScript, and routed using the official Model Context Protocol (`@modelcontextprotocol/sdk/server/index.js`).

At its core, the server bridges the semantic capabilities of Large Language Models (LLMs) with the strict, deterministic environments of Mobile Automation (Node.js, WebdriverIO, Appium, Cucumber-JS).

### Core Components

```text
[ LLM Client ] (Cursor, Claude Desktop, Antigravity)
       │
   (stdio / SSE Transport)
       │
[ Appium MCP Server ] (index.ts)
       │
       ├─► 🛠️ Analysis Layer (CodebaseAnalyzerService)
       ├─► 🧠 Brain Layer (LearningService, TestGenerationService)
       ├─► 💾 I/O & Safety Layer (FileWriterService, SecurityUtils)
       └─► 📱 Execution Layer (ExecutionService, AppiumSessionService)
```

---

## 🔍 The Analysis Layer

Before generating new code, the AI must understand what already exists to prevent duplication.

### `CodebaseAnalyzerService.ts`
We discarded simple regex string-matching in favor of **Abstract Syntax Tree (AST)** parsing using `ts-morph`.
* **Step Definition Extraction**: AST parsing natively understands Given/When/Then annotations, mapping the exact regex patterns and identifying the source file context.
* **POM Method Extraction**: Identifies class names, public methods, and signature types within the `pages/` directory.

### `AuditLocatorService.ts`
Scans existing TypeScript Page Objects. If the LLM previously generated (or a human wrote) brittle `XPath` or generic `CSS` selectors, this service generates an MD-formatted Audit recommending migration to native `accessibility-id` schemas.

## 🧠 The Brain Layer (Generation & Self-Healing)

### `TestGenerationService.ts`
The Orchestrator. When invoked via `generate_cucumber_pom`, it packages a massive context prompt containing:
1. Target App capabilities (`mcp-config.json`).
2. The user's plain-English intent.
3. The JSON analysis generated by `analyze_codebase`.
4. (Optional) Base64 screenshots and Live XML `AppiumSessionService` dumps.
5. Injected Rules from the `LearningService`.

### `SelfHealingService.ts`
If a WebdriverIO test fails due to a `NoSuchElement` error:
1. The error stack trace is extracted.
2. The broken step and exact locator string are identified.
3. The server takes a fresh XML snapshot of the screen via the active `AppiumSessionService`.
4. A context prompt is generated instructing the LLM to map the broken selector to the *current* XML node tree.
5. If the AI returns a valid selector, it can be tested instantly via `verify_selector`.
6. Successful heals are sent to `reportHealSuccess()` to permanently teach the system the new rule.

## 💾 The I/O & Safety Layer

Mobile infrastructure can be destructive. Writing bad files breaks the Node runtime. 

### `SecurityUtils.ts`
Provides three layers of security validations:
1. **Shell Sanitization**: Escapes user/LLM input bound for `exec()` child processes.
2. **Code Auditing**: RegEx safety nets prevent the generation of `eval()`, `exec()`, or `process.env.*` leakages inside the tests.
3. **Secret Scanning**: Feature files are scanned post-generation for hardcoded generic passwords or API keys before writes are permitted.

### `FileWriterService.ts`
Uses a secure staging system:
1. Parses the AI's JSON output (`filesToCreate`, `filesToUpdate`).
2. Runs AST linting.
3. Backs up the *existing* files to `.AppForge/backups/<timestamp>`.
4. Commits writes atomicaly. Supports a `--dryRun` boolean to bypass disk writes entirely.
5. **Cross-Platform Verification**: If `mcp-config.json` is set to `platform: both`, the service rejects POM generation unless the LLM yielded both `Page.android.ts` and `Page.ios.ts`.

---

## 📱 The Execution Layer

### `AppiumSessionService.ts`
Rather than forcing the LLM to blindly guess element locations based solely on `.apk` metadata, this service maintains a long-lived WebdriverIO session hooked into an **active** Android Emulator or iOS Simulator.
* Exposes `getPageSource()` for accurate `accessibility-id` extraction.
* Exposes `takeScreenshot()` for multi-modal vision context.

### `ExecutionService.ts`
Wraps the `cucumber-js` binary. Supports tag filtering (`--tags '@android'`) natively driven by the AI to limit scope, and parses the `cucumber-report.json` to extract actionable metrics (Pass/Fail counts, duration).

## 🛣️ Data Flow: Scenario Generation

To understand the beauty of the orchestration, follow the data flow of a user asking "Add a test for logging out":

1. **User asks AI**: "Write a logout test".
2. **AI calls `analyze_codebase`**: Server returns JSON showing `Given I login as "user"` exists in `login.steps.ts` and `LoginPage.ts` has a `clickProfile()` method.
3. **AI calls `generate_cucumber_pom`**: Server injects the analysis.
4. **AI responds**: With a JSON payload requesting a new `logout.feature`, a new `logout.steps.ts` containing the regex `/I log out/`, and a `ProfilePage.ts` class.
5. **AI calls `validate_and_write`**: 
    - The Server runs `SecurityUtils` audit (Pass).
    - The Server runs Gherkin linting (Pass).
    - The Server runs `tsc --noEmit` (Pass).
    - The Server safely writes the files to disk.
6. **AI calls `run_cucumber_test`**: The server spins up Appium, executes the suite natively, and reports back the success status.
