# AppForge Vendor Security Self-Assessment

This document serves as a comprehensive **Vendor Security Assessment** and **Compliance FAQ** for **AppForge** (Mobile Automation MCP Server). It is intended for enterprise Information Security (InfoSec), IT Procurement, and Data Privacy teams — particularly those in regulated sectors such as Healthcare, Financial Services, and Government — evaluating the tool for internal acquisition or developer adoption.

It covers compliance with **GDPR, HIPAA, PCI-DSS, SOC 2**, and general enterprise vendor security requirements.

---

## 🏗️ 1. Architecture & Data Flow Overview

**Q: Is AppForge a cloud-hosted SaaS platform?**
**A:** No. AppForge is a **locally executed developer tool** (an MCP server binary) that runs entirely on the developer's machine or within your organization's private infrastructure. The tool authors do not operate any central server, cloud database, or hosted API.

**Q: What types of data does AppForge process?**
**A:** AppForge processes the following data categories — all exclusively **locally**, never transmitting to the authors' infrastructure:
- Mobile app binary file paths (`.apk`, `.ipa`, `.app`)
- Appium XML UI Hierarchy dumps (structural layout of app screens)
- Base64 screenshots captured from device/emulator screens
- Test scripts, Cucumber feature files, and Page Object source code
- Appium/WebdriverIO test execution logs
- Credentials stored in local `.env` files (accessed read-only by the `CredentialService`)

**Q: What data is collected or transmitted to the tool authors?**
**A:** **None.** AppForge has zero built-in telemetry, crash-reporting, analytics, or "phone home" functionality. The authors have no visibility into how the tool is being used.

## 🛡️ 2. Regulatory Compliance

**Q: Is AppForge compliant with GDPR?**
**A:** Yes. AppForge's authors are neither a **Data Controller** nor a **Data Processor** under GDPR definitions. No personal data passes to or through the author's infrastructure. Organizations retain complete data sovereignty over all artifacts generated by the tool.

**Q: Is AppForge compliant with HIPAA?**
**A:** Yes, when used correctly. AppForge does not store, transmit, or process Protected Health Information (PHI) to any external party. However, if the mobile application under test contains real patient data on its screens, developers **must** use test environments with synthetic patient data — not production systems with live PHI — to ensure end-to-end HIPAA compliance during test execution. (See Section 3 for the LLM caveat.)

**Q: Is AppForge compliant with PCI-DSS?**
**A:** Yes, with configuration. AppForge does not transmit cardholder data externally. Test scripts and screenshots remain on local storage or private CI infrastructure. Organizations must ensure their Android/iOS emulator environments and any attached cloud device farms are configured within PCI-DSS scope and that screenshots containing card numbers are not retained post-testing.

**Q: Is the tool SOC 2 or ISO 27001 certified?**
**A:** As a locally executable open-source tool rather than a managed cloud service, SOC 2 Type II and ISO 27001 audits (which apply to cloud infrastructure management) are not directly applicable. However, AppForge is architecturally designed to conform to your organization's existing internal SOC 2 boundary — it adds no new external data flows.

## 🤖 3. Third-Party Data Processors (LLMs & Cloud Device Farms)

### 3a. LLM Provider
**Q: Does AppForge send data to third-party AI APIs?**
**A:** AppForge itself does not call any AI API. However, as an MCP (Model Context Protocol) server, it provides structured context — including app XML hierarchies, screenshots, and test logs — to the **LLM client host** configured by the developer (e.g., Claude Desktop, Cursor, or a local Ollama instance).

**Q: Who is the Data Processor for AI interactions?**
**A:** The LLM provider chosen by your organization (e.g., Anthropic, OpenAI, Google, AWS Bedrock). AppForge is not responsible for those providers' privacy practices.

**Q: What must we do to ensure compliance with the LLM side?**
**A:** Enterprise users operating in regulated environments **must**:
1. Use an Enterprise/API tier with their LLM provider.
2. Ensure a signed **Data Processing Agreement (DPA)** or **Business Associate Agreement (BAA)** is in place with the LLM provider (mandatory for HIPAA).
3. Verify the LLM provider enforces a **Zero-Data Retention / Zero-Training policy** on API inputs.
4. Consider using a **locally-hosted LLM** (e.g., Ollama + Llama 3) for maximum data sovereignty, where no data ever leaves the network perimeter.

### 3b. Cloud Device Farms (BrowserStack, Sauce Labs, AWS Device Farm)
**Q: When using cloud device farms, who processes the app binary and test data?**
**A:** If the organization connects AppForge to a cloud device farm (via Appium capabilities pointing to a cloud hub), the **cloud device farm vendor** becomes a Data Processor. Organizations must:
1. Ensure a signed DPA with the cloud device farm vendor.
2. Configure AppForge's `mcp-config.json` to use only the organization-approved device farm endpoints.
3. Never run tests containing real sensitive data (PHI, PII, financial records) against a third-party cloud device farm without a confirmed contractual data isolation guarantee.

## 🔒 4. Security Architecture & Vulnerability Management

**Q: How does the server prevent path traversal or sandbox escapes when writing files?**
**A:** The `FileWriterService` validates every target file path through `SecurityUtils.validateProjectRoot()`, ensuring that all file write operations stay strictly within the declared `projectRoot` directory. Attempts to write to paths like `../../etc/passwd` are blocked at the validation layer before any disk I/O occurs.

**Q: How does AppForge prevent command injection?**
**A:** Before any command arguments (like tags or test paths) are passed to `npx wdio` or similar CLI executables, they are processed by `SecurityUtils.sanitizeForShell()`, which strips shell-dangerous characters (`;`, `&`, `|`, `>`, `<`, `$`).

**Q: What prevents the AI from generating insecure code?**
**A:** Before any AI-generated TypeScript is written to disk, the `validate_and_write` tool performs a **static security audit** of the generated code that detects and blocks:
- `eval()` calls
- `child_process.exec()` usage
- Raw `process.env` leakages directly into output
- Common credential patterns (API keys, hardcoded passwords)

If violations are found, a 🔒 **SECURITY AUDIT FAILURE** is returned to the AI, forcing regeneration using secure practices (e.g., `dotenv` and the `getUser()` helper).

**Q: How is the V8 Sandbox (`execute_sandbox_code`) secured?**
**A:** The sandbox isolates dynamic code execution inside a hardened Node.js `vm.Script` context. The following protections are active:

| Protection | Mechanism |
|---|---|
| **No `eval()` / `new Function()`** | Blocked by static regex scan AND `vm.createContext({ codeGeneration: { strings: false } })` |
| **No `require()` / `import()`** | Undefined in sandbox context + regex pre-scan |
| **No `process` / `globalThis`** | Excluded from context allowlist |
| **No network access** | `fetch` not injected into sandbox context |
| **Timeout enforcement** | Default 10s via `vm.Script.runInContext({ timeout })` |
| **Context isolation** | Fresh `vm.createContext()` per execution — no state leakage between calls |

**Q: How are credentials stored and accessed?**
**A:** Credentials are stored in a local `.env` file, managed exclusively through the dedicated `set_credentials` tool (which applies schema-level validation). Credential files are `.gitignore`d by default in every scaffolded project, preventing accidental commits to source control.

**Q: How are third-party NPM dependencies managed for vulnerabilities?**
**A:** The project uses standard NPM modules. Organizations are strongly encouraged to run SCA (Software Composition Analysis) scans using Snyk, Dependabot, or SonarQube on `package.json` before approving the tool for internal enterprise use.

## ☁️ 5. Cloud & Container Deployments (Docker, AWS, Kubernetes)

AppForge ships with a production-ready **Dockerfile**. When deploying the MCP server in a cloud or containerised environment, the following compliance requirements shift to the **organization's infrastructure** (Shared Responsibility Model):

**Q: How is data residency maintained in the cloud?**
**A:** Any test artifacts generated inside the container (Cucumber reports, XML hierarchies, screenshots) will write to the container's ephemeral storage or mounted volumes (e.g., AWS EBS, EFS, Azure Files). Organizations must ensure:
- Storage volumes are **encrypted at rest** (e.g., using AWS KMS or Azure Key Vault).
- Volumes are provisioned in a **GDPR / HIPAA-compliant hosting region** (e.g., `eu-west-1` for GDPR, a US-based region with a HIPAA-eligible service BAA for HIPAA).

**Q: How should secrets be managed in a container deployment?**
**A:** Never bake credentials or `.env` files into the Docker image. Instead, inject secrets at runtime using:
- **AWS Secrets Manager** or **AWS SSM Parameter Store**
- **HashiCorp Vault**
- **Kubernetes Secrets** (stored as environment variables or mounted volumes)

**Q: How do we secure the network connection to the containerized MCP server?**
**A:** AppForge supports two transport modes:
- **`stdio` (default):** Communicates via standard I/O — no network port is opened. This is the most secure mode.
- **`SSE (Server-Sent Events)` on port 3100:** If using SSE transport, ensure the container is placed inside a **private VPC**, accessed only through a TLS/HTTPS-secured API Gateway or Internal Load Balancer. It must **never** be exposed directly to the public internet.

**Q: Can containers be made ephemeral to minimize data residency risk?**
**A:** Yes. AppForge is fully stateless — it reads the project root on each tool call and writes results back. Containers should be treated as ephemeral: launch for each CI pipeline run, execute tests, export reports to S3/Artifact Storage, and terminate. This minimizes the window during which XML hierarchies or screenshots containing sensitive app data reside in compute memory.

## 📱 6. Mobile-Specific Considerations

**Q: Are app binaries (.apk/.ipa) transmitted externally?**
**A:** No. AppForge only stores the **file system path** to your app binary in `mcp-config.json`. The binary itself is never read, parsed, or transmitted by the MCP server. It is passed directly to your locally running Appium server or a cloud device farm of your choice.

**Q: What about Base64 screenshots passed to the AI?**
**A:** When using the `generate_cucumber_pom` or `self_heal_test` tools with a `screenshotBase64` parameter, the image is included in the prompt sent to your configured LLM. Organizations should ensure:
- Screenshots do not contain real PII, PHI, or financial data (use test accounts with synthetic data).
- The LLM provider has a zero-retention agreement in place.

## ✅ 7. Enterprise Adoption Checklist

Complete this checklist before deploying AppForge in a regulated enterprise environment:

- [ ] Connect AppForge to an Enterprise LLM API with a signed **DPA/BAA** and confirmed **Zero-Data Retention** policy.
- [ ] If using a local LLM (Ollama), verify it is running within your network perimeter.
- [ ] If using a cloud device farm (BrowserStack, LambdaTest), sign a **DPA** with the vendor.
- [ ] Run a SCA vulnerability scan (Snyk / Dependabot) on `package.json` before approving for internal use.
- [ ] In containerized / cloud deployments: encrypt volumes at rest with AWS KMS or equivalent.
- [ ] In containerized / cloud deployments: restrict network access via private VPC; use TLS for SSE transport.
- [ ] Use **synthetic / mock data** (via `generate_test_data_factory`) for all mobile tests — do not test with real patient, financial, or user data.
- [ ] Ensure `.env` files and `users.{env}.json` files containing credentials are `.gitignore`d and not committed to source control.
- [ ] Restrict `projectRoot` to a dedicated test workspace directory with the minimum necessary filesystem permissions.
