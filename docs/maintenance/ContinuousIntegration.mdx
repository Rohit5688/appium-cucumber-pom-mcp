Running AppForge in CI is more involved than web testing because you need an Android emulator or real device. This guide covers two strategies: **Android Emulator** (free, slower) and **Cloud Device Farm** (paid, faster).

---

## 🏗️ CI Architecture for Mobile

```
Checkout → Install Node + Appium → Start Emulator → Build APK → Start Appium Server → Run Tests → Upload Reports
```

:::caution[Real Devices in CI]
Real devices require USB connection or cloud integration (BrowserStack, LambdaTest, Sauce Labs). The emulator approach documented here runs entirely on the CI runner.
:::

## GitHub Actions — Android Emulator

### Generate a Workflow File

```
"Generate a GitHub Actions CI workflow for my AppForge Android project running @smoke tests on push"
```

### Full Android Emulator Workflow

```yaml
name: AppForge Android Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  android-e2e:
    name: Android @smoke Tests
    runs-on: macos-latest  # macOS has hardware acceleration for emulators
    timeout-minutes: 60

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Setup Java
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'

      - name: Setup Android SDK
        uses: android-actions/setup-android@v3

      - name: Install AVD (Android Virtual Device)
        run: |
          sdkmanager --install "system-images;android-33;google_apis;arm64-v8a"
          sdkmanager --install "platform-tools" "platforms;android-33"
          avdmanager create avd -n ci_device -k "system-images;android-33;google_apis;arm64-v8a" --force
        env:
          ANDROID_SDK_ROOT: ${{ env.ANDROID_SDK_ROOT }}

      - name: Start Android Emulator
        run: |
          $ANDROID_HOME/emulator/emulator -avd ci_device -no-window -no-audio -no-snapshot &
          adb wait-for-device
          # Wait for device to fully boot
          until adb shell getprop sys.boot_completed 2>/dev/null | grep -m 1 "1"; do sleep 5; done
          echo "Emulator ready"

      - name: Install Node dependencies
        run: npm ci

      - name: Install Appium + Drivers
        run: |
          npm install -g appium
          appium driver install uiautomator2

      - name: Start Appium Server
        run: |
          appium --base-path /wd/hub --log appium.log &
          sleep 5

      - name: Run @smoke tests
        run: npx wdio run wdio.conf.ts --suite smoke
        env:
          APPIUM_URL: http://127.0.0.1:4723/wd/hub

      - name: Upload test artifacts
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: appforge-results-${{ github.run_number }}
          path: |
            mcp-logs/
            appium.log
          retention-days: 14
```

:::tip[Use `macos-latest` for Android]
GitHub's `ubuntu-latest` runners cannot run the Android emulator with KVM acceleration unless you use nested virtualization (experimental). `macos-latest` runners support hardware-accelerated emulators natively.
:::

## GitHub Actions — BrowserStack App Automate

For faster, more stable CI with real devices:

```yaml
name: AppForge on BrowserStack

on:
  push:
    branches: [main]

jobs:
  browserstack:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Upload APK to BrowserStack
        id: upload
        run: |
          RESPONSE=$(curl -u "$BS_USER:$BS_KEY" \
            -X POST "https://api-cloud.browserstack.com/app-automate/upload" \
            -F "file=@./builds/app-debug.apk")
          echo "app_url=$(echo $RESPONSE | jq -r '.app_url')" >> $GITHUB_OUTPUT
        env:
          BS_USER: ${{ secrets.BROWSERSTACK_USERNAME }}
          BS_KEY: ${{ secrets.BROWSERSTACK_ACCESS_KEY }}

      - name: Run tests on BrowserStack
        run: npx wdio run wdio.browserstack.conf.ts
        env:
          BROWSERSTACK_USERNAME: ${{ secrets.BROWSERSTACK_USERNAME }}
          BROWSERSTACK_ACCESS_KEY: ${{ secrets.BROWSERSTACK_ACCESS_KEY }}
          BS_APP_URL: ${{ steps.upload.outputs.app_url }}
```

## GitLab CI

```yaml
image: ubuntu:22.04

stages:
  - setup
  - test

variables:
  ANDROID_SDK_VERSION: "33"

before_script:
  - apt-get update && apt-get install -y nodejs npm openjdk-17-jre-headless
  - npm ci
  - npm install -g appium
  - appium driver install uiautomator2

android:smoke:
  stage: test
  script:
    - appium --base-path /wd/hub &
    - sleep 5
    - npx wdio run wdio.conf.ts --suite smoke
  artifacts:
    when: always
    paths:
      - mcp-logs/
    expire_in: 14 days
  only:
    - main
    - merge_requests
```

## 🔐 Environment Variables in CI

| `mcp-config.json` field | CI Secret | Notes |
| :--- | :--- | :--- |
| `appiumUrl` | `APPIUM_URL` | Override in pipeline env if Appium runs on a different port |
| `credentials` user passwords | `TEST_PASSWORD` | Map via `manage_config set_credentials` in CI setup step |
| BrowserStack username | `BROWSERSTACK_USERNAME` | Cloud device farm auth |
| BrowserStack key | `BROWSERSTACK_ACCESS_KEY` | Never hardcode |

## 📊 Reports & Artifacts

### Forge Session Viewer

Upload `mcp-logs/viewer.html` as an artifact — it renders in any browser without a server:

```yaml
- uses: actions/upload-artifact@v4
  with:
    name: forge-session-viewer-${{ github.run_number }}
    path: mcp-logs/viewer.html
    retention-days: 30
```

### Appium Log

Always upload `appium.log` on failures — it contains the device error stack that explains why tests failed at the Appium level:

```yaml
- name: Upload Appium log on failure
  if: failure()
  uses: actions/upload-artifact@v4
  with:
    name: appium-log-${{ github.run_number }}
    path: appium.log
```

### Cucumber JSON Report

Located at `mcp-logs/cucumber-report.json`. Feed it into Allure or Cucumber Reports service:

```bash
npx allure generate mcp-logs/cucumber-report.json --clean -o allure-report
```

## 🔁 Async Test Polling

For long regression suites, use AppForge's async job system:

```bash
# Start tests asynchronously
JOB_ID=$(npx appforge run android --async --tags @regression | jq -r '.jobId')

# Poll every 30 seconds (max 20 minutes)
for i in $(seq 1 40); do
  STATUS=$(npx appforge status $JOB_ID | jq -r '.status')
  echo "Attempt $i: $STATUS"
  [ "$STATUS" = "completed" ] && break
  [ "$STATUS" = "failed" ] && exit 1
  sleep 30
done
```

## ✅ CI Readiness Checklist

- [ ] `.apk` either committed or fetched from artifact store
- [ ] Java 17+ installed on runner (`setup-java` action)
- [ ] Android SDK installed and `ANDROID_HOME` set
- [ ] Emulator fully booted before Appium starts (use `adb wait-for-device`)
- [ ] Appium server started with `--base-path /wd/hub` and given 5s to initialize
- [ ] `npm ci` (not `npm install`) for reproducible lockfile
- [ ] All secrets in CI vault — never in `mcp-config.json`
- [ ] `timeout-minutes` set to 60+ for emulator runs (boot alone is 2–5 min)
- [ ] Artifacts uploaded with `if: always()`
