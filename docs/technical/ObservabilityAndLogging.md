# 📊 Observability & System Metrics

AppForge includes built-in observability mechanisms designed to keep AI loops informed, provide structured telemetry, and prevent hanging sessions in high-concurrency or CI/CD environments.

---

## 📋 1. The Unified Logger

AppForge utilizes a domains-aware `Logger` utility to normalize system outputs. This makes it easier to track errors during live debug sessions and when aggregating CI logs.

### Key Features:
- **Domain Prefixing**: Every entry is prefixed with its associated component (e.g., `[AppForge] [SessionManager]`).
- **Structured Error Tracing**: standardizes JavaScript Error tracking to provide pure, actionable stack traces without ambiguous console dumps.
- **Log Levels**: Support for `DEBUG`, `INFO`, `WARN`, and `ERROR` to filter noise in production environments.

> [!NOTE]
> When investigating a failed task, always check the `[AppForge] [*]` prefixed logs in your terminal output for a high-level summary of the failure root cause.

---

## 📈 2. Concurrency: `SessionManager`

For large-scale automation, AppForge intelligently manages multiple parallel tasks via the `SessionManager` singleton.

- **Project-Scoped Pooling**: Connects WebdriverIO instances scoped to your `projectRoot`. This allows multiple repositories on the same host to run without session collisions.
- **Persistence & Heartbeats**: Tracks active sessions and detects "abandoned" connections (e.g., if an LLM crashes mid-generation), automatically terminating them to free up device ports.

---

## 📊 3. Health & Metrics Tools

### `get_session_health`
AI agents and developers can call this tool to perform a diagnostic check on the active AppForge environment.

**Metrics Provided:**
- **Uptime & Memory**: Basic server health metrics.
- **Active Sessions**: Count of active vs. idle driver connections.
- **Device Locking**: Identifies which `projectRoot` currently owns specific physical or virtual device locks.

---

## 🛡️ 4. Monitoring Best Practices

- **CI Integration**: In CI environments, set `LOG_LEVEL=info` to capture the `[AppForge] [RunCucumber]` lifecycle events.
- **Post-Run Cleanup**: Always ensure your CI pipeline calls `end_appium_session` or uses the `hooks.ts` generated by `setup_project` to guarantee session teardown.
- **Metric Audits**: If you encounter port-busy errors, use `get_session_health` to identify which processes are holding locks before manually killing the Appium server.
