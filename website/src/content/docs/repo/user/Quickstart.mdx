## 🏗️ Phase 1: Preparation (1 Minute)

Ensure your environment is ready. AppForge works best when it can 'see' your local files and your Appium server.

<Steps>

1.  **Project Context**  
    Open your terminal in an empty directory or your existing WebdriverIO/Appium project.
    
2.  **Connect the MCP**  
    Ensure AppForge is added to your Claude Desktop or Cursor MCP settings.  
    `{ "appforge": { "command": "npx", "args": ["-y", "@page-agent/appforge-mcp"] } }`

3.  **The "Hello" Prompt**  
    Ask the agent:  
    > *"Is AppForge ready? Perform a structural audit and initialize a new project if empty."*

</Steps>

---

## ✍️ Phase 2: Create Your First Test (2 Minutes)

We'll use a real-world example: **Testing a Mobile Login Flow**.

<Steps>

1.  **Direct the Architect**  
    Provide the target app path (e.g., `.apk` or `.app`) and the goal.  
    > *"Start an Appium session using the 'pixel8' profile. Navigate to the Login screen. Generate a BDD test for a successful login. Then verify the 'Welcome' toast message appears."*

2.  **Review the Blueprint**  
    The AI will use `inspect_ui_hierarchy` to find stable Accessibility IDs and then generate:
    - `features/login.feature`
    - `pages/LoginPage.ts` (with typed gesture methods)
    - `step-definitions/login.steps.ts`

3.  **Atomic Writing**  
    Once you approve the code, tell the agent:  
    > *"Write these files to disk and run a validation check."*  
    *The agent will use `validate_and_write` to ensure everything follows POM standards.*

</Steps>

---

## 🏃 Phase 3: Run & Auto-Heal (2 Minutes)

The true power of AppForge is in handling mobile UI churn autonomously.

<Steps>

1.  **Trigger Execution**  
    > *"Run the login test and give me a summary of the results."*

2.  **Simulate UI Drift**  
    Imagine a developer changed the login button ID. If the test fails, simply say:  
    > *"The login test failed with a NoSuchElement exception. Use `self_heal_test` to fix it."*

3.  **Witness the Repair**  
    The MCP will re-scan the live device XML, find the new selector via **Fuzzy Semantic Matching**, and update your Page Object file automatically.

</Steps>

## 📂 What Just Happened?

AppForge created a standards-compliant, enterprise-grade structure for you:

<FileTree>
- features/
  - **login.feature** (The human-readable spec)
- pages/
  - **BasePage.ts** (The framework foundation)
  - **LoginPage.ts** (Your app-specific logic)
- step-definitions/
  - **login.steps.ts** (The automation glue)
- utils/
  - **MobileGestures.ts** (High-level gesture drivers)
- **mcp-config.json** (The AI's operational brain)
</FileTree>

## 🚧 Troubleshooting the "First 5 Minutes"

| Issue | Likely Cause | Fix 🛠️ |
| :--- | :--- | :--- |
| **"Appium not found"** | Server not running. | Run `appium --base-path /wd/hub` in a separate terminal. |
| **"Profile missing"** | Config not initialized. | Check `capabilitiesProfiles` in your `mcp-config.json`. |
| **"Brittle Locators"** | Brittle XML structure. | Use `audit_mobile_locators` to see health scores. |

## 🧠 Knowledge Audit

<details>
<summary>**Q1: What happens if the healed selector is also brittle?**</summary>
AppForge calculates a **Confidence Score** for every healing candidate. If the score is low, it will prompt the user to manually verify the element before committing the patch to the Page Object.
</details>

<details>
<summary>**Q2: Can I use this for both Android and iOS?**</summary>
**Answer**: Yes. By using the `BasePage` pattern and abstracting gestures, you can write a single Gherkin scenario that drives both platforms using different `capabilitiesProfiles`.
</details>
